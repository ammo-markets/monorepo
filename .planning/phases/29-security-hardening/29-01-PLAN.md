    **SEC-03: Fix rate limiter IP extraction in middleware**

    In `apps/web/middleware.ts`, fix `getRateLimitKey()`:

    Current (INSECURE -- trusts client-supplied first entry):
    ```
    return forwarded.split(",")[0]!.trim();
    ```

    Change to use the LAST entry (closest trusted proxy):
    ```
    const parts = forwarded.split(",");
    return parts[parts.length - 1]!.trim();
    ```

    The last entry in x-forwarded-for is added by the trusted reverse proxy (Vercel) and cannot be spoofed by the client. The first entry is client-supplied and trivially forgeable.

    **SEC-05: Add domain, URI, and chainId verification to SIWE**

    In `apps/web/app/api/auth/verify/route.ts`, expand the `siweMessage.verify()` call to also check domain, URI, and chainId.

    Import `avalancheFuji` from `viem/chains`.

    After parsing the SIWE message but before calling verify, extract expected values:
    ```typescript
    const expectedDomain = process.env.NEXT_PUBLIC_APP_DOMAIN ?? "localhost:3000";
    const expectedUri = process.env.NEXT_PUBLIC_APP_URL ?? "http://localhost:3000";
    const expectedChainId = avalancheFuji.id; // 43113
    ```

    Then pass these to verify:
    ```typescript
    const result = await siweMessage.verify({
      signature: body.signature,
      nonce: session.nonce,
      domain: expectedDomain,
      scheme: expectedUri.startsWith("https") ? "https" : "http",
    });
    ```

    Additionally, after verify succeeds, explicitly check chainId:
    ```typescript
    if (result.data.chainId !== expectedChainId) {
      return Response.json(
        { error: "Invalid chain ID" },

    }
    ```

    ```typescript
    if (result.data.domain !== expectedDomain) {
      return Response.json(
        { error: "Invalid domain" },
        { status: 401 },
      );
    }
    ```

    Note: The `siwe` library's verify() accepts `domain` and `scheme` options to validate the message's domain and URI scheme. Combined with explicit chainId checking post-verify, this enforces full SIWE policy -- not just nonce validity.

    **SEC-02: Make useKycSubmit throw on non-2xx responses**

    In `apps/web/hooks/use-kyc.ts`, fix the `useKycSubmit` mutation function:

    Current (BROKEN -- swallows errors):
    ```typescript
    mutationFn: async (data: KycFormData) => {
      const res = await fetch("/api/users/kyc", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(data),
      });
      return res.json();
    },
    ```

    Change to throw on non-2xx so TanStack Query's `onError` / `isError` / error boundaries activate:
    ```typescript
    mutationFn: async (data: KycFormData) => {
      const res = await fetch("/api/users/kyc", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(data),
      });
      if (!res.ok) {
        const errorData = await res.json().catch(() => ({}));
        throw new Error(
          (errorData as { error?: string }).error ?? `KYC submission failed (${res.status})`
        );
      }
      return res.json();
    },
    ```

    **Also fix useKycStatus query to throw on non-2xx:**

    Current (swallows all errors, returns fallback):
    ```typescript
    const res = await fetch("/api/users/kyc");
    const data = await res.json();
    ```

    Change to check res.ok:
    ```typescript
    const res = await fetch("/api/users/kyc");
    if (!res.ok) {
      throw new Error(`Failed to fetch KYC status (${res.status})`);
    }
    const data = await res.json();
    ```

    Keep the outer catch block that returns `{ kycStatus: "NONE", kycPrefill: undefined }` -- this is the fallback for unauthenticated users. But remove the try/catch wrapping the entire function so that actual server errors (500s) bubble up through TanStack Query's error handling rather than being silently swallowed. Actually, the catch block is fine for graceful degradation on auth failures (401) but the mutation MUST throw. The query can keep its catch for graceful fallback since a 401 on GET just means "not logged in yet" which is expected.

    Final approach for query: Only add the `res.ok` check but keep the catch. This way server errors (500) throw, but network failures still degrade gracefully.
