<task type="auto">
  <name>Task 1: Add ProtocolStats, ActivityLog, and UserPreference models to Prisma schema</name>
  <files>packages/db/prisma/schema.prisma</files>
  <action>
Add three new models to schema.prisma after the existing BlockCursor model:

**1. ProtocolStats** (DB-01) -- Per-caliber aggregate metrics, one row per caliber:
- `id` String @id @default(cuid())
- `caliber` Caliber @unique -- one stats row per caliber
- `totalMinted` String @default("0") -- total minted amount (use String like Order.amount for large numbers)
- `totalRedeemed` String @default("0") -- total redeemed amount
- `netSupply` String @default("0") -- minted minus redeemed
- `userCount` Int @default(0) -- distinct users who traded this caliber
- `computedAt` DateTime @default(now()) -- when stats were last computed (internal bookkeeping)
- Add `@@index([caliber])` (though @unique already creates one, explicit for clarity)

**2. ActivityLog** (DB-02) -- One row per completed transaction, structured data only:
- `id` String @id @default(cuid())
- `type` OrderType -- MINT or REDEEM (reuse existing enum; no SWAP yet since swap is not in the protocol)
- `caliber` Caliber
- `amount` String -- round amount
- `txHash` String? -- on-chain transaction hash
- `walletAddress` String -- user who performed the transaction
- `createdAt` DateTime @default(now())
- Add indexes: `@@index([walletAddress])`, `@@index([createdAt])`, `@@index([type])`

**IMPORTANT: Do NOT include a `status` field on ActivityLog.** Per user decision "Final state only", only completed transactions are stored in this table. A status field is redundant since every row is implicitly COMPLETED. The existing Order table retains status for in-progress tracking; ActivityLog is the finalized record.

**3. UserPreference** (DB-03) -- Favorite calibers per user, created on first use:
- `id` String @id @default(cuid())
- `userId` String @unique -- one preferences row per user (not created until first favorite)
- `favoriteCalibers` Caliber[] -- Postgres array of Caliber enum values
- `createdAt` DateTime @default(now())
- `updatedAt` DateTime @updatedAt
- `user` User @relation(fields: [userId], references: [id])
- Add the reverse relation on User model: `preferences UserPreference?`

Per user decisions:
- Stats are per-caliber only (no global aggregates, no time series)
- Activity log stores structured data only (type, caliber, amount, txHash as fields; UI builds display strings)
- Activity log records final state only (one row per completed transaction -- NO status field)
- UserPreference stores favorite calibers only (no display settings, no default trade settings)
- UserPreference is toggle-based and created on first use (no row until user explicitly favorites)
