  <name>Task 1: Create stats computation module with explicit backfill and wire into worker cron</name>
  <files>apps/worker/src/stats.ts, apps/worker/src/index.ts, apps/worker/src/lib/constants.ts</files>
  <action>
**Create `apps/worker/src/stats.ts`:**

Export TWO async functions:

**1. `backfillActivityLog()`** -- One-time backfill, runs on worker startup:
1. Check if ActivityLog table is empty: `const count = await prisma.activityLog.count()`
2. If `count > 0`, log `[backfill] ActivityLog already populated (${count} rows), skipping` and return early
3. If empty, query all completed orders: `prisma.order.findMany({ where: { status: "COMPLETED" } })`
4. For each completed order, create an ActivityLog row: `prisma.activityLog.create({ data: { type: order.type, caliber: order.caliber, amount: order.amount, txHash: order.txHash, walletAddress: order.walletAddress, createdAt: order.updatedAt } })`. Use `createMany` if possible for efficiency, or batch with `prisma.$transaction`.
5. Log: `[backfill] Backfilled ${orders.length} completed orders into ActivityLog`
6. On error: log `[backfill] Failed to backfill ActivityLog:` with the error. Do NOT throw -- worker should continue running.

**2. `computeStats()`** -- Periodic stats recomputation:
1. Queries the Order table to compute per-caliber aggregates from completed orders:
   - For each of the 4 calibers (NINE_MM, FIVE_FIVE_SIX, TWENTY_TWO_LR, THREE_OH_EIGHT):
     - `totalMinted`: SUM of amount WHERE type=MINT AND status=COMPLETED
     - `totalRedeemed`: SUM of amount WHERE type=REDEEM AND status=COMPLETED
     - `netSupply`: totalMinted - totalRedeemed
     - `userCount`: COUNT DISTINCT walletAddress WHERE status=COMPLETED
   - Use Prisma `$queryRaw` with `CAST(amount AS NUMERIC)` for SUM, or fetch all completed orders and aggregate in JS (simpler, sufficient for testnet scale).
2. Upsert each caliber's ProtocolStats row with the computed values.
3. Log success: `[stats] Computed protocol stats for 4 calibers`
4. On error: log the error with `[stats] Failed to compute stats:` and return (do NOT throw -- caller keeps running). Per user decision: serve stale data on failure, log error, retry next cycle.

**Update `apps/worker/src/lib/constants.ts`:**
Add `export const STATS_INTERVAL_MS = 15 * 60 * 1000;` (15 minutes per user decision).

**Update `apps/worker/src/index.ts`:**
After the existing polling interval setup, add stats and backfill:
1. Import `computeStats` and `backfillActivityLog` from `./stats`
2. Import `STATS_INTERVAL_MS` from `./lib/constants`
3. Run `await backfillActivityLog()` once on startup (before stats computation, after existing initialization)
4. Run `await computeStats()` once on startup (after backfill completes)
5. Set up `setInterval(computeStats, STATS_INTERVAL_MS)` alongside the existing event polling interval
6. Clear both intervals in the shutdown handler
7. Log: `[worker] Computing stats every ${STATS_INTERVAL_MS / 60000} minutes`

The stats cron and backfill run independently of the event indexer polling loop. They share the same process but do not block each other.

Per user decisions:
- Data source is DB only (no on-chain reads) -- aggregate from already-indexed orders
- Recompute every 15 minutes
- On failure: serve stale data, log the error, retry next cycle
