  <name>Task 1: Create stats computation module and wire into worker cron</name>
  <files>apps/worker/src/stats.ts, apps/worker/src/index.ts, apps/worker/src/lib/constants.ts</files>
  <action>
**Create `apps/worker/src/stats.ts`:**

Export an async function `computeStats()` that:
1. Queries the Order table to compute per-caliber aggregates from completed orders:
   - For each of the 4 calibers (NINE_MM, FIVE_FIVE_SIX, TWENTY_TWO_LR, THREE_OH_EIGHT):
     - `totalMinted`: SUM of amount WHERE type=MINT AND status=COMPLETED
     - `totalRedeemed`: SUM of amount WHERE type=REDEEM AND status=COMPLETED
     - `netSupply`: totalMinted - totalRedeemed
     - `userCount`: COUNT DISTINCT walletAddress WHERE status=COMPLETED
   - Use Prisma `groupBy` or raw aggregation queries as needed. Since `amount` is stored as String, use Prisma `$queryRaw` with `CAST(amount AS NUMERIC)` for SUM, or fetch all completed orders and aggregate in JS (simpler, sufficient for testnet scale).
2. Upsert each caliber's ProtocolStats row with the computed values.
3. Log success: `[stats] Computed protocol stats for 4 calibers`
4. On error: log the error with `[stats] Failed to compute stats:` and return (do NOT throw -- caller keeps running). Per user decision: serve stale data on failure, log error, retry next cycle.

**Update `apps/worker/src/lib/constants.ts`:**
Add `export const STATS_INTERVAL_MS = 15 * 60 * 1000;` (15 minutes per user decision).

**Update `apps/worker/src/index.ts`:**
After the existing polling interval setup, add a second `setInterval` for stats:
1. Import `computeStats` from `./stats`
2. Import `STATS_INTERVAL_MS` from `./lib/constants`
3. Run `computeStats()` once on startup (after backfill completes, before polling starts)
4. Set up `setInterval(computeStats, STATS_INTERVAL_MS)` alongside the existing event polling interval
5. Clear both intervals in the shutdown handler
6. Log: `[worker] Computing stats every ${STATS_INTERVAL_MS / 60000} minutes`

The stats cron runs independently of the event indexer polling loop. They share the same process but do not block each other.

Per user decisions:
- Data source is DB only (no on-chain reads) -- aggregate from already-indexed orders
- Recompute every 15 minutes
- On failure: serve stale data, log the error, retry next cycle

**Create `apps/web/app/api/user/preferences/route.ts`** -- Authenticated endpoint:

Follow the same patterns as `apps/web/app/api/users/profile/route.ts` for auth and error handling.

**GET handler:**
1. Call `requireSession()` to get authenticated user
2. Query `prisma.userPreference.findUnique({ where: { userId } })` using the user's ID (look up user by `session.address.toLowerCase()` first, same as profile route)
3. If no preferences row exists, return `{ favoriteCalibers: [] }` (per user decision: created on first use, not pre-populated)
4. If exists, return `{ favoriteCalibers: [...] }` with calibers mapped via PRISMA_TO_CALIBER

**PUT handler:**
1. Call `requireSession()` to get authenticated user
2. Parse JSON body, validate with zod: `{ favoriteCalibers: z.array(z.enum(["9MM", "556", "22LR", "308"])) }`
3. Convert display calibers to Prisma enum values using the reverse mapping (e.g., "9MM" -> "NINE_MM"). Use `CALIBER_TO_PRISMA` from shared package (or build reverse map inline).
4. Upsert: `prisma.userPreference.upsert({ where: { userId }, create: { userId, favoriteCalibers: [...] }, update: { favoriteCalibers: [...] } })`
5. Return the updated preferences
6. Per user decision: toggle-based favorites, created on first use via upsert

Use `import { requireSession } from "@/lib/auth"` and `import { prisma } from "@ammo-exchange/db"`.

Check if `CALIBER_TO_PRISMA` exists in shared package. If not, create the reverse mapping inline or add it to the shared package as a counterpart to `PRISMA_TO_CALIBER`.

Per user decision: GET/PUT /api/user/preferences, standalone endpoint separate from user profile, favorite calibers only.
