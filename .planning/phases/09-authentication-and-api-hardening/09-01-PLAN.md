    1. Install dependencies in apps/web:
       `pnpm --filter @ammo-exchange/web add siwe iron-session`
       - siwe: SIWE message creation and verification
       - iron-session: Encrypted cookie-based sessions (stateless, no DB needed)

    2. Create `apps/web/lib/auth.ts`:
       - Define IronSession data interface: `{ nonce: string; siwe?: { address: string; chainId: number } }`
       - Export `sessionOptions` for iron-session: cookie name "ammo_session", password from `process.env.SESSION_SECRET` (32+ char secret), cookie options (httpOnly, secure in production, sameSite "lax", maxAge 86400 = 24 hours)
       - Export `getSession(): Promise<IronSessionData>` -- uses `cookies()` from `next/headers` internally: `getIronSession<SessionData>(await cookies(), sessionOptions)`. No request parameter needed.
       - Export `requireSession(): Promise<{ address: string; chainId: number }>` -- calls `getSession()` and throws/returns 401 Response if no siwe data. No request parameter. This is the helper all protected routes will use.
       - Export `requireKeeper(): Promise<{ address: string; chainId: number }>` -- calls `requireSession()`, then reads the AmmoManager contract's `isKeeper(address)` using the server-side viem publicClient from `@/lib/viem`. Returns 403 if not a keeper. Import AmmoManagerAbi from `@ammo-exchange/contracts/abis` and CONTRACT_ADDRESSES from `@ammo-exchange/shared`. No request parameter.
       - These functions use `cookies()` from `next/headers` internally -- this is the modern Next.js 15 App Router pattern for iron-session v8+. Do NOT accept a request parameter.

    3. Create `apps/web/app/api/auth/nonce/route.ts`:
       - GET handler: generate a random nonce via `generateNonce()` from siwe package
       - Store nonce in session: `session.nonce = nonce; await session.save();`
       - Return `{ nonce }` as JSON

    4. Create `apps/web/app/api/auth/verify/route.ts`:
       - POST handler: parse body with `{ message: string, signature: string }`
       - Create SiweMessage from message string, call `.verify({ signature, nonce: session.nonce })`
       - Validate nonce matches session nonce (prevent replay)
       - On success: store `session.siwe = { address: siweMessage.address.toLowerCase(), chainId: siweMessage.chainId }`, clear nonce, save session
       - Also upsert the user in the database here (combines auth + registration into one atomic step): `prisma.user.upsert({ where: { walletAddress: address }, create: { walletAddress: address }, update: {} })`
       - Return `{ address, chainId }`
       - On failure: return 401 with error message

    5. Create `apps/web/app/api/auth/session/route.ts`:
       - GET handler: read session, return `{ address, chainId }` if authenticated, or `{ address: null }` if not

    6. Create `apps/web/app/api/auth/logout/route.ts`:
       - POST handler: destroy session (`session.destroy()`), return `{ ok: true }`

    7. Update `CLAUDE.md` Environment Variables section to add:
       ```
       SESSION_SECRET=xxxxxxxx...        # 32+ character secret for iron-session cookie encryption (required)
       ALLOWED_ORIGINS=https://...       # Comma-separated allowed origins for CORS (optional, defaults to http://localhost:3000)
       ```
       Add these two lines to the existing env vars block, after the FUJI_RPC_URL line.

  <action>
    1. Create `apps/web/hooks/use-siwe.ts`:
       - Export `useSiwe()` hook that manages SIWE authentication state
       - State: `{ isSignedIn: boolean; isSigningIn: boolean; address: string | null; signIn: () => Promise<void>; signOut: () => Promise<void>; checkSession: () => Promise<void> }`
       - `checkSession`: GET /api/auth/session, update state from response
       - `signIn`:
         a. GET /api/auth/nonce to get nonce
         b. Construct SIWE message using `SiweMessage` from siwe: domain = window.location.host, uri = window.location.origin, version "1", chainId from wagmi useChainId, address from useAccount, nonce from step a, statement "Sign in to Ammo Exchange"
         c. Use wagmi's `useSignMessage` to sign the message (call `signMessageAsync({ message: siweMessage.prepareMessage() })`)
         d. POST /api/auth/verify with { message: siweMessage.prepareMessage(), signature }
         e. On success, update state to signed in
       - `signOut`: POST /api/auth/logout, update state
       - On mount, call checkSession to restore session from cookie
       - On account change (useAccount address changes), if signed in and address differs from session address, call signOut (session invalidation on wallet switch)
       - Use `import type { SiweMessage as SiweMessageType }` for type-only import, and regular import for the class constructor

    2. Update `apps/web/hooks/use-wallet.ts`:
       - Remove the `registerWallet` function entirely (registration is now handled server-side in /api/auth/verify)
       - Remove the registeredRef, registrationFailed state, retryRegistration callback, and the registration useEffect
       - Keep the rest of the hook unchanged (connect, disconnect, switchToFuji, isWrongNetwork, etc.)
       - Remove `registrationFailed` and `retryRegistration` from the return object

    3. Fix ERR-02 race condition in `apps/web/app/api/users/register/route.ts`:
       - The current upsert approach is already idempotent, but the real race condition is rapid connect/disconnect/reconnect causing multiple concurrent upserts
       - Wrap the upsert in a Prisma interactive transaction with serializable isolation: `prisma.$transaction(async (tx) => { ... }, { isolationLevel: "Serializable" })`
       - This prevents duplicate user creation even under concurrent requests
       - NOTE: This endpoint remains for backward compatibility (worker fallback) but is no longer called from the frontend

    4. Check if any component references `registrationFailed` or `retryRegistration` from useWallet. If so, remove those references. Search the codebase for these strings.
