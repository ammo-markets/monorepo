  <action>
    Import `requireSession` and `requireKeeper` from `@/lib/auth`. Apply auth to each route as follows:

    **User routes (requireSession -- returns 401 if no session):**

    1. `GET /api/orders` -- Call `requireSession(req)` at top. If it returns a Response (401), return it. Otherwise, use the returned `address` instead of the `wallet` query param. Remove the wallet query param entirely -- the session IS the identity. Update the Prisma query to use `session.address` instead of `parsed.data.wallet`.

    2. `GET /api/orders/[id]` -- Call `requireSession(req)`. After fetching the order, verify `order.walletAddress === session.address` (users can only view their own orders). Return 403 if mismatch.

    3. `GET /api/balances` -- Call `requireSession(req)`. Use `session.address` instead of the `wallet` query param. Remove wallet query param validation.

    4. `POST /api/redeem/shipping` -- Call `requireSession(req)`. After fetching the order, verify `order.walletAddress === session.address` (AUTH-05: caller owns the order). Return 403 with "Not your order" if mismatch.

    5. `GET /api/users/kyc` -- Call `requireSession(req)`. Use `session.address` instead of the `wallet` query param.

    6. `POST /api/users/kyc` -- Call `requireSession(req)`. Use `session.address` instead of body wallet. Add testnet gate (AUTH-04): check `process.env.NODE_ENV !== "production"`. If production, return 403 "KYC auto-approve disabled in production". Remove wallet from body validation.

    **Admin routes (requireKeeper -- returns 401 if no session, 403 if not keeper):**

    7. `GET /api/admin/orders` -- Call `requireKeeper(req)` at top. If it returns a Response (401/403), return it. Rest of handler unchanged.

    8. `GET /api/admin/stats` -- Call `requireKeeper(req)`. Rest unchanged.

    **Public routes (no auth needed):**

    9. `GET /api/market` -- No auth. Market data is public. Leave unchanged.
    10. `GET /api/activity` -- No auth. Recent activity is public. Leave unchanged.

    **Pattern for requireSession/requireKeeper in route handlers:**
    ```typescript
    const session = await requireSession();
    if (session instanceof Response) return session;
    // session is now { address: string; chainId: number }
    ```
    NOTE: The exact pattern depends on how requireSession was implemented in Plan 01. Read `apps/web/lib/auth.ts` first to match the actual API. If requireSession throws instead of returning Response, use try/catch.

    **Important:** When removing the `wallet` query param from routes, also remove the zod schema validation for it (the wallet identity now comes from the session, not the request).

<task type="auto">
  <name>Task 2: Add Next.js middleware for CORS and rate limiting</name>
  <files>
    apps/web/middleware.ts

  <action>
    Create `apps/web/middleware.ts` with:

    1. **CORS handling:**
       - Define `ALLOWED_ORIGINS`: read from `process.env.ALLOWED_ORIGINS` (comma-separated) or default to `["http://localhost:3000"]` for development
       - On every API request, check the `Origin` header against ALLOWED_ORIGINS
       - If Origin is present and NOT in allowed list, return 403 "Origin not allowed"
       - If Origin is allowed, set response headers: `Access-Control-Allow-Origin: <origin>`, `Access-Control-Allow-Methods: GET, POST, OPTIONS`, `Access-Control-Allow-Headers: Content-Type`, `Access-Control-Allow-Credentials: true`
       - Handle OPTIONS preflight requests: return 204 with CORS headers
       - NOTE: Do NOT block requests with no Origin header (server-to-server, curl, etc. don't send Origin). Only block requests where Origin IS present but not in the allowed list.

    2. **Rate limiting (in-memory, simple):**
       - Use a Map<string, { count: number; resetAt: number }> for tracking
       - Key: IP address from `request.headers.get("x-forwarded-for")` or `request.ip` or "unknown"
       - Window: 60 seconds, limit: 100 requests per window per IP
       - On limit exceeded: return 429 "Too Many Requests" with `Retry-After` header
       - Periodically clean stale entries (every 1000 requests, delete entries where resetAt < Date.now())
       - NOTE: This is an in-memory rate limiter. It works for single-instance deployments (Vercel serverless functions don't share memory, so it's per-instance). For true rate limiting on Vercel, you'd use Vercel KV or Upstash Redis. This is sufficient for testnet -- document this limitation in a code comment.

    3. **Middleware config:**
       - Export `config` with matcher: `"/api/:path*"` (only run on API routes)
       - Do NOT match auth routes for CORS (they need to work for the sign-in flow) -- actually, auth routes still need CORS, so match all API routes
       - Do NOT apply rate limiting to GET /api/market or GET /api/activity (public, read-only, may be called frequently) -- actually, still rate limit them but with a higher threshold. Simplest approach: use the same 100/min limit for all routes (good enough for testnet).

    4. **Middleware function:**
       ```
       export function middleware(request: NextRequest) {
         // 1. CORS check
         // 2. Rate limit check
         // 3. Return NextResponse.next() with CORS headers attached
       }
       ```
