  <action>
Update the Order model in `packages/db/prisma/schema.prisma`:

1. **Add `logIndex` column**: `logIndex Int` (required, non-nullable). Every on-chain event has a logIndex.

2. **Rename `amount` to `usdcAmount`**: Change the existing `amount String` field to `usdcAmount String?` (nullable). Mint creation sets this; redeem creation leaves it null.

3. **Make `tokenAmount` nullable**: The existing `tokenAmount String?` stays nullable (already is). Redeem creation sets this; mint creation leaves it null.

4. **Replace unique constraint**: Remove `@unique` from `txHash` field. Add composite unique: `@@unique([txHash, logIndex])` to the Order model. Keep `txHash` as `String?` (nullable, since existing data may not have it -- but new orders always will).

5. **Drop old `amount` column**: The old `amount` column is replaced by `usdcAmount`. Remove it entirely.

6. **Keep existing indexes**: Retain `@@index([txHash])` for query performance. Add `@@index([logIndex])` is NOT needed (composite unique handles it).

The final Order model field list should be:
```
id              String      @id @default(cuid())
userId          String
type            OrderType
status          OrderStatus @default(PENDING)
caliber         Caliber
usdcAmount      String?     // Raw USDC wei (set on mint creation, set on redeem finalization)
tokenAmount     String?     // Raw token wei (set on redeem creation, set on mint finalization)
onChainOrderId  String?
walletAddress   String?
txHash          String?
logIndex        Int
chainId         Int
createdAt       DateTime    @default(now())
updatedAt       DateTime    @updatedAt

user            User             @relation(fields: [userId], references: [id])
shippingAddress ShippingAddress?

@@unique([txHash, logIndex])
@@index([userId])
@@index([txHash])
@@index([status])
@@index([walletAddress])
@@index([onChainOrderId])
```

After editing the schema, run the migration. Per user decision: wipe existing order data (Fuji testnet, no preservation needed). Use `pnpm db:migrate` with the `--name` flag for a descriptive migration name. If there are data conflicts (the `amount` column rename), use `prisma migrate dev --create-only` first, then edit the migration SQL to handle the rename, then apply.

**Important**: Since we are wiping data, the simplest approach is:
1. Edit schema.prisma with the changes above
2. Run: `cd packages/db && npx prisma migrate dev --name composite_order_uniqueness_and_amounts`
3. If Prisma warns about data loss (dropping `amount` column), confirm it is acceptable
4. Run: `pnpm db:generate` to regenerate the Prisma client

After migration, verify by running `pnpm --filter @ammo-exchange/db check` (or the package's typecheck) to confirm the generated client compiles.
