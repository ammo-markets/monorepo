Add RPC resilience (retry with backoff), chain reorg protection, startup environment validation, and graceful shutdown that drains in-flight work.

Purpose: The worker currently crashes on any RPC failure, has no reorg awareness (shallow reorgs can cause missed events or phantom data), starts without checking env vars (confusing runtime errors), and calls process.exit(0) during shutdown without waiting for the current poll to finish (risking partial transaction commits).

Output: A production-hardened worker that retries transient failures, handles reorgs safely, fails fast on misconfiguration, and shuts down cleanly.
</objective>

**Part A: Add reorg protection to `apps/worker/src/indexer.ts`**

Modify `pollOnce()` to implement a confirmation window rollback:

1. Import `CONFIRMATION_BLOCKS` from `./lib/constants`.

2. After computing `rawFrom` (cursor + 1), subtract the confirmation window:

   ```typescript
   // Roll back cursor by CONFIRMATION_BLOCKS to re-process recent blocks.
   // This catches shallow reorgs where events may have been added or removed.
   // Idempotent handlers (txHash-based upsert) make re-processing safe.
   const reorgSafeFrom =
     rawFrom > CONFIRMATION_BLOCKS ? rawFrom - CONFIRMATION_BLOCKS : rawFrom;
   const fromBlock =
     reorgSafeFrom < DEPLOYMENT_BLOCK ? DEPLOYMENT_BLOCK : reorgSafeFrom;
   ```

3. The existing handlers are already idempotent (txHash-based upsert for creates, updateMany for status changes), so re-processing the same events is a no-op. No other changes needed to handlers.

4. Only advance the cursor to `batchEnd` (not `batchEnd - CONFIRMATION_BLOCKS`). The rollback happens on the NEXT poll -- we always re-scan the last N blocks to catch any reorged events.

**Part B: Update `apps/worker/src/index.ts` for graceful shutdown**

Replace the current shutdown logic that calls `process.exit(0)` immediately:

1. Import `env` from `./lib/env` at the TOP of the file (before any other imports that might use env vars). This ensures validation runs first. The import itself triggers the validation side effect.

2. Add a shutdown coordination mechanism:

   ```typescript
   let isShuttingDown = false;
   let currentPoll: Promise<void> | null = null;
   ```

3. In the polling interval callback, track the current poll:

   ```typescript
   const intervalId = setInterval(async () => {
     if (isProcessing || isShuttingDown) return;
     isProcessing = true;
     try {
       currentPoll = pollOnce();
       await currentPoll;
     } catch (error) {
       console.error("[worker] Polling error (will retry):", error);
     } finally {
       currentPoll = null;
       isProcessing = false;
     }
   }, POLL_INTERVAL_MS);
   ```

4. Replace the shutdown handler:

   ```typescript
   const shutdown = async () => {
     if (isShuttingDown) return; // Prevent double-shutdown
     isShuttingDown = true;
     console.log("[worker] Shutting down -- draining in-flight work...");
     clearInterval(intervalId);

     // Wait for current poll cycle to complete
     if (currentPoll) {
       try {
         await currentPoll;
         console.log("[worker] In-flight poll completed.");
       } catch (error) {
         console.error(
           "[worker] In-flight poll failed during shutdown:",
           error,
         );
       }
     }

     console.log("[worker] Shutdown complete.");
     process.exit(0);
   };

   process.on("SIGTERM", shutdown);
   process.on("SIGINT", shutdown);
   ```

5. Also wrap the initial backfill in a shutdown check -- if SIGTERM arrives during backfill, it should still complete the current batch before exiting:

   console.log("[worker] Running initial backfill...");
   await pollOnce();
   if (isShuttingDown) {
     console.log("[worker] Shutdown requested during backfill. Exiting.");
     process.exit(0);
   }
