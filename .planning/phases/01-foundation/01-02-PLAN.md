```typescript
// Bidirectional mapping: Prisma Caliber enum <-> shared Caliber type
// NOTE: Prisma enum values use NINE_MM, FIVE_FIVE_SIX, etc. because Prisma enum
// identifiers cannot start with a digit (e.g., "9MM" is invalid). This is a Prisma
// limitation, not a naming preference. The mapping bridges the two representations.
export const PRISMA_TO_CALIBER = {
  NINE_MM: "9MM",
  FIVE_FIVE_SIX: "556",
  TWENTY_TWO_LR: "22LR",
  THREE_OH_EIGHT: "308",
} as const satisfies Record<string, Caliber>;

export const CALIBER_TO_PRISMA = {
  "9MM": "NINE_MM",
  "556": "FIVE_FIVE_SIX",
  "22LR": "TWENTY_TWO_LR",
  "308": "THREE_OH_EIGHT",
} as const satisfies Record<Caliber, string>;
```

The `satisfies` ensures type safety without narrowing the type. The Prisma enum values (NINE_MM etc.) stay as-is because Prisma enum identifiers cannot start with a digit — renaming to "9MM" etc. is not possible. The bidirectional mapping bridges Prisma's naming constraint with the shared Caliber type.

Modify the existing `packages/db/prisma/schema.prisma`:

**Order model changes:**

1. Add `onChainOrderId String?` field — nullable because orders exist in the DB before on-chain confirmation. Place it after `amount`.
2. Add `walletAddress String?` field — nullable for backward compatibility but will be populated by the worker from event data. Place it after `onChainOrderId`. This field enables querying orders by wallet without joining through User.
3. Add new indexes:
   - `@@index([walletAddress])` — for querying orders by wallet
   - `@@index([onChainOrderId])` — for looking up orders by on-chain ID
   - Keep existing indexes: `@@index([userId])`, `@@index([txHash])`, `@@index([status])`

**New BlockCursor model** (add after the Order model):

```prisma
model BlockCursor {
  id              String   @id @default(cuid())
  contractAddress String   @unique
  chainId         Int
  lastBlock       BigInt   @default(0)
  updatedAt       DateTime @updatedAt

  @@index([contractAddress])
}
```

Fields:

- `id`: cuid primary key (consistent with other models)
- `contractAddress`: unique — one cursor per CaliberMarket contract. The worker will create 4 rows (one per caliber market).
- `chainId`: Int — identifies the chain (43113 for Fuji, 43114 for mainnet)
- `lastBlock`: BigInt with default 0 — the last block number the worker has processed for this contract. BigInt is correct here because it stays in the database and worker (never serialized to JSON via Next.js API routes).
- `updatedAt`: auto-updated timestamp

**Do NOT rename the Caliber enum values.** Keep NINE_MM, FIVE_FIVE_SIX, TWENTY_TWO_LR, THREE_OH_EIGHT as-is. Prisma enum identifiers cannot start with a digit, so "9MM" etc. are not valid Prisma enum values. The mapping is handled in the shared package (Task 1).
